\begin{abstract}
Commercial search APIs are the de facto retrieval layer for tool-using LLM agents, yet they are typically evaluated as static ranked-list components. We argue that a search API is better understood as a \emph{decision surface}: its snippets, rankings, and URL pool jointly shape whether an agent searches again, opens a page, answers from snippets, or abstains. We test this view on the 100-query \textsc{SealQA-Hard} subset by running the same GPT-5.4 agent with a two-tool interface (\texttt{search\_web}, \texttt{fetch\_page}) over three production providers: Brave, Tavily, and Firecrawl. An LLM judge labels every visible URL across 6{,}909 surface judgments (2{,}202 / 2{,}481 / 2{,}226 per provider) as gold-supporting, contradicting, supporting-the-model-answer, or garbage, enabling a per-query oracle of which URLs the agent \emph{should} have opened. The three providers produce nearly identical exact-match accuracy (21, 21, 23 / 100) but achieve it through fundamentally different retrieval pipelines: Brave's surface lifts the URL-pool ceiling to 33\% via rich \texttt{extra\_snippet} blocks; Tavily concentrates 48\% of gold-supporting URLs at rank one, raising the fetch-recall of supporting URLs to 31\%; Firecrawl exposes the sparsest surface but compensates with broad, mostly-blind fetching that wins 20\% of no-gold-visible queries. Fetch policy ROI varies from $+26$ percentage points (Tavily) to $-13$pp (Firecrawl) on the same model. We use the contradict-to-gold URL ratio (Brave $0.92$, Tavily $1.87$, Firecrawl $2.59$) as an answer-independent surface-quality measurement and argue that retrieval-budget evaluation, not top-$k$ relevance, is the right axis for assessing search providers for agentic systems.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Search-augmented LLM agents now ship in consumer products, deep-research assistants, and enterprise workflows. They all share the same architecture: a model calls a search API, reads results, optionally opens pages, and answers. The search API is treated as a fungible commodity: swap Brave for Tavily and you supposedly get the same agent. This paper argues that view is wrong, but \emph{not in the way prior work has measured}. Final answer accuracy varies surprisingly little across providers (Section~\ref{sec:results}). What varies dramatically is how the agent allocates its retrieval budget: which URLs it opens, which it ignores, when it searches again, and how often it answers correctly from snippets alone. These behavioral differences are larger and more practically important than the accuracy delta.

We frame search APIs as \emph{decision surfaces}. The provider's snippets, extra-snippet blocks, URL pool, and ranking order are the only signals an agent sees before committing fetch tokens. Two providers with similar top-$k$ URL relevance can induce completely different fetch policies: one rewards reading the top result, another rewards reading lower-ranked results, a third rewards skipping fetches entirely. Final accuracy can mask this because the underlying model adapts -- but the cost, latency, and failure-mode distribution do not.

Our contribution is fourfold:

\begin{enumerate}
    \item \textbf{Agentic retrieval evaluation with a per-URL oracle.} We instrument an agent with separate \texttt{search\_web} and \texttt{fetch\_page} tools, then run an LLM-judge oracle over every visible URL, whether the model opened it or not (\S\ref{sec:methodology}). This yields a per-query confusion matrix of \smart, \missed, \blind, and \nofetch{} decisions for each of three commercial providers.
    \item \textbf{Empirical mechanism analysis across Brave, Tavily, and Firecrawl} on the 100-query \textsc{SealQA-Hard} subset, totaling 6{,}909 surface judgments (6{,}519 snippet-only rows plus 350 page-visible rows) and 300 query-provider trajectories (\S\ref{sec:results}).
    \item \textbf{Three findings that survive scrutiny}: providers achieve indistinguishable accuracy through different pipelines (\S\ref{sec:finding-pipelines}); fetch-policy return-on-investment is provider-dependent (\S\ref{sec:finding-roi}); and per-URL contradict-to-gold ratio is a clean, answer-independent surface-quality metric that varies 2.8$\times$ across providers (\S\ref{sec:finding-ratio}).
    \item \textbf{Actionable design guidance}: a fixed agent fetch policy is suboptimal across providers, and the right policy can be predicted from surface-level signals before any model run (\S\ref{sec:discussion}).
\end{enumerate}

We deliberately avoid framing this as a provider leaderboard. The leaderboard view (``provider $A$ beats provider $B$ on EM'') would conclude there is little to choose between Brave, Tavily, and Firecrawl on hard QA. The decision-surface view, by contrast, gives a structural reason \emph{why} accuracy converges and identifies the policy and cost levers that providers actually move.

\section{Related Work}
\label{sec:relatedwork}

\paragraph{Search-augmented and agentic QA.} Retrieval-augmented generation \citep{lewis-etal-2020-retrieval, izacard-grave-2021-leveraging} and search-augmented agents \citep{shuster2022blenderbot, schick-etal-2023-toolformer, yao2023react} have largely measured end-to-end answer quality. BrowseComp \citep{browsecomp2025} stresses agent capability on hard browsing tasks. Our work shares the hard-QA setting but inverts the focus: the controlled variable is the search provider, not the agent.

\paragraph{Retrieval evaluation and judge oracles.} Top-$k$ relevance metrics \citep{thakur-etal-2021-beir} dominate retriever evaluation but capture only the static URL ranking, not the agent's decisions. LLM-as-judge methodology \citep{zheng2023judging} has been used to grade answers; we extend it to grade every visible URL, producing an oracle of which results an ideal agent would have opened. To our knowledge, the smart/foolish/missed-skip matrix this enables has not previously been instantiated across commercial providers with a fully judged URL pool.

\paragraph{Hard QA benchmarks.} \textsc{SealQA-Hard} \citep{sealqa2025} contains 254 web-search questions designed to be unsolvable from parametric memory and to expose conflicting or unhelpful retrieved evidence. We use its publicly released subset, sampled to 100 queries via deterministic stratified sampling on \texttt{freshness}, \texttt{search\_results}, and \texttt{topic} (see Appendix~\ref{app:dataset}).

\paragraph{Commercial search APIs as research artifacts.} Brave Search \citep{bravesearch-docs}, Tavily \citep{tavily-docs}, and Firecrawl \citep{firecrawl-docs} are widely used in agent stacks but have not, to our knowledge, been compared as decision surfaces under a controlled agentic protocol. Existing comparisons report top-$k$ URL relevance or downstream EM only.

\subsection{Agent Architecture}

The agent exposes two tools.

\paragraph{\texttt{search\_web(query)}} returns up to ten ranked results per provider. Each result carries title, URL, domain, snippet, and any provider-native \texttt{extra\_snippet} blocks. Crucially, no page content is included; the agent sees only what the provider's API natively returns.

\paragraph{\texttt{fetch\_page(document\_id)}} fetches and extracts the full text of a previously-surfaced URL via Jina Reader \citep{jina-reader}. The model receives extracted markdown, fetch status, extractor name, and character count. The fetch backend is identical across providers, so any differences in fetched content are attributable to the URL pool, not the extractor.

At each turn the model may search again, fetch any previously-seen URL, answer with \texttt{FINAL ANSWER: $\langle x\rangle$}, or abstain. The maximum iteration budget is 10, the maximum results per search is 10, and the model is GPT-5.4 (temperature 0, max output tokens 4096). The system prompt, user-query wrapper, and tool schema are identical across providers; only the search backend changes.

\subsection{Providers and Dataset}

We use the v2 endpoints of three production providers: Brave Web Search, Tavily Search, and Firecrawl Search (configuration in Appendix~\ref{app:provider-config}). Every provider is configured with \texttt{country=US}, English language settings, and no provider-side page scraping; we want to evaluate the search surface, not the provider's optional fetch layer. The exact HTTP request payloads we sent are reproduced in Appendix~\ref{app:provider-payloads}.

The dataset is 100 queries from \textsc{SealQA-Hard} \cite{sealqa2025}, selected by deterministic proportional stratified sampling over \texttt{freshness} $\times$ \texttt{search\_results} (SealQA's ``conflicting / unhelpful'' label) $\times$ \texttt{topic} (seed 20260509). The sample is 57\% \texttt{conflicting} and 43\% \texttt{unhelpful}; covers all six SealQA topics (Science \& Tech 27, Sports 22, Entertainment 22, Other 12, Politics 9, History \& Geography 8); 43\% \texttt{slow-changing}, 32\% \texttt{never-changing}, 25\% \texttt{fast-changing}; and 61\% from before-2024, 16\% 2024, 23\% 2025. The worst stratum drift versus the 254-row source distribution is $+3.3$pp. Full per-cell allocations and the multi-label question-type breakdown are in Appendix~\ref{app:dataset}.

\subsection{Per-URL Oracle Judging}
\label{sec:judging}

For every URL the agent saw, whether or not it opened it, we run an LLM judge (Kimi-K2.6, temperature 0). The judge inspects exactly the same XML fragment the agent saw at decision time (title, URL, domain, snippet, \texttt{extra\_snippet}s if any), plus the gold answer, the question, and the model's final answer. The schema is

\begin{small}
\begin{verbatim}
{
  "contains_gold_answer": bool,
  "supports_model_answer": bool,
  "contradicts_gold_answer": bool,
  "is_garbage": bool,
  "answer_span": str,
  "confidence": float
}
\end{verbatim}
\end{small}

URLs the agent fetched are judged twice: once on the \emph{snippet-only} surface (matching what the agent saw before deciding to fetch) and once on the \emph{page-visible} surface (matching what the agent saw after fetching). This yields two ceilings: the snippet-surface ceiling (what was answerable before fetching) and the URL-pool ceiling (what was answerable after fetching from any URL in the pool). Total judgments: 6{,}909 across providers (2{,}202 / 2{,}481 / 2{,}226), with 6{,}869 producing a valid structured judgment; the 40 invalid rows are timeouts or schema-parse failures excluded from analysis (Section~\ref{sec:limitations}).

A measurement caveat must be stated up front: Brave's snippet-only surface includes \emph{multiple} \texttt{extra\_snippet} blocks per result (median $\approx$ 3 blocks of 200--400 chars each), while Tavily and Firecrawl expose a single snippet only. This is a provider-design choice, not an artifact of our pipeline, and the agent and the judge see exactly the same text. But it means any judgment on the snippet surface for Brave is made on $\approx$5$\times$ more text per URL than for Tavily or Firecrawl. We report the per-URL contradict-to-gold ratio (Section~\ref{sec:finding-ratio}) as the answer-independent surface metric not subject to this confound (both fields use the same per-URL surface).

\subsection{The Smart / Missed / Blind Matrix}

We partition the 100 queries per provider into four mutually exclusive, exhaustive buckets:

\begin{itemize}
    \item \smart: $\geq$1 gold-supporting URL was visible \emph{and} the model fetched at least one gold-supporting URL.
    \item \missed: $\geq$1 gold-supporting URL was visible \emph{and} the model fetched none of them (possibly fetched something else, possibly fetched nothing).
    \item \blind: no gold-supporting URL was visible \emph{and} the model fetched $\geq$1 URL.
    \item \nofetch: no gold-supporting URL was visible \emph{and} the model fetched none.
\end{itemize}

Throughout, ``gold-supporting URL visible'' means the LLM judge labeled at least one URL in the surface (snippet or fetched page) as \texttt{contains\_gold\_answer}. \smart{} and \missed{} share the same denominator (queries with gold visible); their EM gap is the cleanest empirical estimate of the value of fetching when gold is on the surface.

\centering
\includegraphics[width=0.92\textwidth]{figures/fig1_buckets.pdf}
\caption{Smart / Missed / Blind / No-op partition of the 100 queries per provider, with EM\% inside each bucket. Table~\ref{tab:pipelines} shows three orthogonal pipelines.

\paragraph{Brave: rich snippet surface, broad URL pool, low fetch recall.} Brave's URL-pool ceiling is 33 queries -- nearly double the others. Of those 33, the agent answers 12 correctly (36\%), and an additional 9 correct answers come from queries the judge marked unsupported, for 21 EM in total. Brave returns 87 unique supporting URLs across the 33 ceiling-supported queries ($\approx$2.6 supporting URLs per query on average) but the agent fetches only 9 of them (10.3\% recall). Brave's gold-supporting URLs are roughly uniformly distributed across all 10 ranks (12 at rank 1, 17 at rank 3, 9 at rank 9; see Figure~\ref{fig:rank}).

\paragraph{Tavily: top-rank concentration, high fetch recall, high conditional capture.} Tavily places \textbf{48\% of its gold-supporting URLs at rank 1} (15/31), versus 12\% for Brave and 11\% for Firecrawl. The agent's natural ``open the top result'' tendency therefore converts into 30.8\% fetch recall, three times Brave's. Among the 24 ceiling-supported queries, Tavily's agent captures 12 (50\%), the highest conditional capture rate of any provider. The remaining 9 correct answers come from beyond-ceiling queries, giving 21 EM total.

\paragraph{Firecrawl: sparse surface, broad fetching, beyond-ceiling wins.} Firecrawl's URL-pool ceiling is only 19 queries; the agent captures 7 of them (37\%). Yet Firecrawl reaches 23 EM total -- the highest of the three -- because 16 of those 23 correct answers come from queries the judge marked unsupported. This ``beyond-ceiling EM'' (20\% on the 81 no-gold-visible queries) is double the rate of Brave and Tavily (each 12--13\%). Firecrawl's agent fetches on 81\% of queries and 54\% of those fetches go to rank 1, but most rank-1 URLs do not contain gold by snippet. Wins on the beyond-ceiling queries come from broadly fetching URLs whose snippets were sparse but whose page content held the answer.

The three providers thus converge on $\sim$21--23 EM through (i) reading rich snippets, (ii) reading rank 1, and (iii) reading broadly: three retrieval styles with very different cost and failure-mode profiles. Figure~\ref{fig:buckets} visualizes the resulting smart/missed/blind/no-op partition.

\begin{itemize}
    \item \textbf{Tavily: $+26$pp.} Opening Tavily's top result, when it is gold-supporting, is highly valuable.
    \item \textbf{Brave: $+2$pp.} Brave's rich snippet surface already conveys the answer; the page adds little.
    \item \textbf{Firecrawl: $-13$pp.} On Firecrawl, fetching the gold page actually correlates with \emph{lower} EM than skipping it. The sample is small ($n{=}7$ \smart{}) but the direction is consistent with Firecrawl's beyond-ceiling EM in \blind{}: the model does best on Firecrawl by fetching broadly into the URL pool, not by trusting the snippet-judged signal.
\end{itemize}

The same generic agent policy of ``search, optionally fetch a result, answer'' has positive, near-zero, and negative ROI across the three providers. This is a structural fact about how each provider's surface aligns with the model's natural decision heuristics.

\subsection{Finding 3: A clean, answer-independent surface-quality metric}
\label{sec:finding-ratio}

The headline contains a circularity hazard: a URL labeled \texttt{supports\_model\_answer $\wedge\,\neg$gold} can only exist when the model's answer differed from gold, making any ``misleading content'' count partly tautological with EM. We therefore use the answer-independent \texttt{contradicts\_gold\_answer} field as the surface-quality metric of record. Per URL $u$, with $\mathrm{contradicts}(u)$ and $\mathrm{containsGold}(u)$ the judge's two binary labels, we define the contradict-to-gold ratio

\[
  r_{\text{c:g}} \;=\;
  \frac{|\{u : \mathrm{contradicts}(u)\}|}
       {|\{u : \mathrm{containsGold}(u)\}|}
\]

aggregated across all snippet-judged URLs visible to a provider's agent. $r_{\text{c:g}}$ measures how often the surface delivers contradicting content relative to supporting content, independent of whether the model used either.

\section{Case Studies}
\label{sec:cases}

Two queries exemplify the pipelines (a third is in Appendix~\ref{app:case-c}).

\paragraph{Case A: Tavily's smart fetch wins (\textsc{sealhard\_6f8f541710fe}).}\textit{According to 2021 world population data, how many continents account for less than 10\% of the world's population? Gold = 5.} Brave surfaced Statista, Wikipedia World Population, and Worldometer; the agent fetched Worldometer's population-by-region page and answered \emph{4} (wrong). Tavily surfaced StatisticsTimes (rank 1), Facebook (rank 2), and Wikipedia ``List of continents and continental subregions by population'' (rank 3). The agent fetched the Wikipedia list page and answered \emph{5} (correct). Firecrawl placed the same Wikipedia page at rank 1 but the agent did not fetch, answering \emph{4} from snippets (wrong). The canonical page existed in all three result sets; Tavily's rank ordering induced the right fetch.

\paragraph{Case B: Brave answers without fetching (\textsc{sealhard\_5668b93dd088}).}\textit{Who was the second non-Asian men's soccer player to pass 100 international goals\dots? Gold = Lionel Messi.} Brave's two searches surfaced multiple Wikipedia pages with \texttt{extra\_snippet} blocks naming Messi as a recent 100-goal passer. Zero fetches; answered \emph{Lionel Messi} (correct). Tavily's single search returned a Tehran Times article whose snippet read ``Ronaldo becomes second player to reach 100 international goals''; with no \texttt{extra\_snippet} disambiguation, the agent answered \emph{Cristiano Ronaldo} (wrong). Firecrawl ran four searches and one fetch to reach the correct answer at higher cost. Brave's snippet richness substituted for fetching.

\section{Discussion}
\label{sec:discussion}

\subsection{The right fetch policy is provider-dependent}

A widely-used heuristic in agent stacks is ``fetch the top-3 results.'' Our results show this is provider-dependent in three ways. On Tavily, fetching rank 1 is high-ROI (+26pp on gold-visible queries) but fetching ranks 2--3 may add contradicting context (the 67\%$\to$22\% collapse in Table~\ref{tab:gold-contradict-cross}). On Brave, fetching is largely wasted token cost; the snippet surface already conveys what the model needs in $\geq$36\% of gold-visible cases without any fetch. On Firecrawl, fetching is structurally cheap insurance: the snippet surface under-reports the URL pool's actual support, and broad fetching recovers queries the agent would otherwise lose.

This suggests an \emph{adaptive} policy: condition fetch budget on provider identity (cheap heuristic) or on prompt-visible surface signals (more general). The contradict-to-gold ratio at the per-provider level is a strong correlate of optimal fetch breadth.

\subsection{Why does accuracy converge?}

Two compensating mechanisms explain why EM is similar across very different surfaces. \textbf{Snippet richness substitutes for fetching}: Brave's \texttt{extra\_snippet} blocks let the model answer from the surface alone on a large fraction of gold-visible queries (Case B). \textbf{Broad fetching substitutes for snippet richness}: Firecrawl's sparse snippets force broad fetching (81\% of queries), recovering queries the snippet judge cannot evaluate. The same agent ``compensates its way'' to similar EM in multiple regimes; if you remove richness, the model fetches more, and if you remove fetch budget, the rich-surface providers cope but the sparse-surface ones collapse.

\subsection{Implications for evaluation}

Provider evaluation should report at minimum: (i) per-URL contradict-to-gold ratio; (ii) rank distribution of supporting URLs; (iii) the smart/missed/blind matrix on a fixed agent; (iv) cost per supported answer. Top-$k$ relevance, the dominant retrieval metric today, is insufficient because it ignores how an agent will actually use the surface.

\section{Limitations}
\label{sec:limitations}

\paragraph{The URL-pool ceiling is a lower bound on true URL-pool support.} Of the 6{,}869 valid judgments, only 350 \texttt{page\_visible} rows (101 / 125 / 124 for Brave / Tavily / Firecrawl) received a page-text judgment; the remaining 6{,}519 snippet-only rows were judged on title + URL + snippet (+ \texttt{extra\_snippet}s for Brave) alone. A URL whose snippet was sparse but whose underlying page contained gold can only be discovered as gold-supporting if the model happened to fetch it. A full page-level oracle over all 5{,}565 unique visible URLs (1{,}700 / 2{,}035 / 1{,}830 per provider) is left to future work; Firecrawl's beyond-ceiling EM (Section~\ref{sec:finding-pipelines}) is direct evidence that the current ceiling understates true URL-pool support.

\paragraph{Single judge model.} All 6{,}869 valid judgments come from Kimi-K2.6 at temperature 0. We did not run a second-judge calibration. The contradict-to-gold ratio and the \smart{}/\missed{}/\blind{} matrix are robust to judge bias in the sense that the same judge is applied uniformly to all providers, but absolute ceiling rates may be conservative or liberal.

\paragraph{Single agent model.} All trajectories are GPT-5.4. Provider-specific findings (e.g., Tavily's rank-1 ROI) may shift with a different model. We selected GPT-5.4 because it is a current-generation production model widely used in tool-use deployments.

\paragraph{Observational, not causal.} Surface state and query difficulty co-vary; we cannot claim that the surface \emph{caused} the trajectory differences without a replay experiment in which the same agent state is fed a different provider's results. We leave such counterfactual replays to future work.

\paragraph{Sample size.} 100 queries per provider yields Wilson 95\% CIs of $\pm$8pp for proportions near 0.5. Several cell-level differences in Table~\ref{tab:buckets} do not reach significance at $n{=}100$. The qualitative pattern (Tavily$+26$, Brave$+2$, Firecrawl$-13$) is robust; the precise magnitudes are not.

\paragraph{40 invalid judgments} (6 Brave, 17 Tavily, 17 Firecrawl), comprising timeouts or JSON schema-parse failures, were excluded from analysis. These are roughly uniformly distributed across queries and unlikely to bias provider comparisons.

\paragraph{Cost is the binding constraint on scale.} The full study consumed roughly 17.0M agent tokens across 300 trajectories, plus 6{,}909 LLM-judge calls, plus three commercial search-API plans and a Jina Reader subscription. End-to-end this was the limiting factor for scaling to more queries, more providers, or more agent models; 100 queries is the largest pool we could fully judge under our budget. Releasing the trace and judgment dataset (Section~\ref{sec:resources}) reduces this marginal cost for follow-up work.

\paragraph{Three commercial providers only.} Google Custom Search, Bing, Exa, and Serper are not evaluated. The proposed methodology (snippet+page oracle, contradict:gold ratio) generalizes; we expect similar across-pipeline divergence with other providers.

\section{Conclusion}

Commercial search APIs are usually treated as interchangeable input components for retrieval-augmented agents. We instead treat them as decision surfaces that reshape how the agent allocates its retrieval budget. On 100 hard \textsc{SealQA-Hard} queries, three production providers converge on near-identical EM (21--23 / 100) through three structurally different pipelines (rich snippets, top-rank concentration, broad fetching), fetch-policy ROI varies from $+26$ to $-13$ pp, and per-URL contradict-to-gold ratios vary 2.8$\times$. Retrieval-budget evaluation, not top-$k$ relevance or end-to-end accuracy alone, is the right axis for choosing and routing search providers for agentic systems.

You are an assistant that answers

  results with document_id, title,
  URL, snippet.

</workflow>

<output_format>

</output_format>
\end{verbatim}

\section{Case C: Same page, different counts}
\label{app:case-c}

\textit{Q: How many times in history has the PDC World Darts Championship final featured two English players? Gold = 14.} (\textsc{sealhard\_370de35828d1}.) All three providers placed the Wikipedia ``PDC World Darts Championship'' page at rank 1, and the agent fetched exactly that URL in every provider's run. Yet the final answers were \emph{15} (Brave), \emph{15} (Tavily), and \emph{14} (Firecrawl, correct). The fetched evidence was identical; the surrounding rank-2/3 results and snippets differed (Brave: \texttt{dartsnews.com}; Tavily: \texttt{dartskins.co.uk}; Firecrawl: \texttt{darts501.com}). This case demonstrates that the surface continues to influence reasoning \emph{after} fetching: the rest of the prompt context changes how the model counts entries in the same Wikipedia table.
